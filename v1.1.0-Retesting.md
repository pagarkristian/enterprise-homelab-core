## 3. Vulnerability Analysis (Pre-Hardening)
Initial reconnaissance and penetration testing identified the following critical security flaws:

* **SQL Injection**: Authentication bypass possible on the administrative login portal.
* **Unrestricted File Upload**: Remote Code Execution (RCE) vector via the document upload endpoint.
* **Weak Authentication**: SSH service susceptible to dictionary-based brute force attacks.
* **Insecure Permissions**: Web directory (/uploads) configured with world-writable (777) permissions.

## 4. Remediation & Hardening Report

Following the identification of critical vulnerabilities, the following security patches were successfully implemented:

| Vulnerability / Vector | Remediation Action | Security Patch / Control | Status |
| :--- | :--- | :--- | :--- |
| **SQL Injection** | Implemented Prepared Statements and parameterized inputs for all database queries. | Code Logic Patch (`index.php`) | ![](https://img.shields.io/badge/Status-Fixed-brightgreen?style=flat-square) |
| **File Upload RCE** | Applied server-side whitelist validation for file extensions and disabled script engines. | Access Control List & `.htaccess` | ![](https://img.shields.io/badge/Status-Fixed-brightgreen?style=flat-square) |

## 5. Retesting & Validation (Post-Hardening)
Post-remediation testing was conducted to verify that all attack vectors were successfully neutralized.

### A. File Upload Security Test
* **Objective:** Ensure the newly implemented server-side extension whitelist mechanism effectively blocks malicious runtime payloads (e.g., weaponized `.php` shells) from being written to the web root.
* **Execution Vector:** Attempted to re-upload the `exploit.php` payload through the document submission framework.

> 🚫 **System Security Enforcement Result:**
> The server backend successfully intercepted the anomalous traffic sequence and rejected the file upload, returning an explicit defensive error notification: 
> `"Gagal! Format file tidak diizinkan."`


### B. Brute Force Mitigation Test
* **Objective:** Validate the defensive effectiveness of the Fail2Ban Intrusion Prevention System (IPS) against automated, multi-threaded SSH brute-force attempts simulated via Hydra.
* **Execution Vector:** Initiated an aggressive password spraying sequence targeting Port `2222` from the Auditor Node.

> 🚫 **System Security Enforcement Result:**
> Fail2Ban successfully monitored the active authentication log stream (`/var/log/auth.log`). Upon detecting the threshold breach (3 failed attempts), the daemon automatically triggered an iptables/nftables rule to inject the attacker's IP address into the real-time drop blocklist.

### C. Integrity & Access Control Test
* **Objective:** Ensure the entire deployment environment is thoroughly purged of historical exploitation artifacts and that current access control structures maintain the Principle of Least Privilege (PoLP).
* **Execution Vector:** Audited the filesystem layout and verified individual directory privilege masks across critical paths.

> 🛡️ **System Security Enforcement Result:**
> All historical unauthenticated web shells (`exploit.php`) were completely eradicated from the web root. File access control checkouts confirm that directory write privileges are restricted strictly to authorized maintenance identities, and public upload directories have binary script engine parsing features disabled.
