##  1. Network Topology & Asset Inventory 🌐

The laboratory environment is orchestrated within an isolated internal network segment (`10.216.27.0/24`) to simulate a Medium-Sized Enterprise (SMB) DMZ zone without direct external internet exposure to maintain safety boundaries.

### 📋 IP Address Allocation Matrix

| Host Asset | Role / Function | Operating System | IP Address | Active Open Ports |
| :--- | :--- | :--- | :--- | :--- |
| **Core-Gateway** | Edge Router / NAT | MikroTik RouterOS v7.21.4 | `10.216.27.1` | Port 8291 (Winbox) |
| **TechSecure-Portal** | Production Target Web | Ubuntu Server 24.04 LTS | `10.216.27.100` | Port 80 (HTTP), Port 21 (FTP), Ports 139/445 (Samba) |
| **Auditor-Node** | Offensive Weapon Base | Kali Linux (Pre-built VM) | `10.216.27.10` | *Dynamic / Outbound Only* |


##  Network Adapter Configuration
We use **2 Network Adapters** on every Operating System (MikroTik, Ubuntu Server, Kali Linux):
* **Adapter 1 (NAT):** Used to provide internet access to our virtual machines so they can download system updates and required packages.
* **Adapter 2 (Host-Only Adapter):** Used to allocate local IP addresses so all operating systems can interconnect within the same private virtual environment.

##  Mikrotik Configuration 
We need to ensure that the MikroTik gateway can forward traffic to the internet using the following configuration parameters:
* **Chain:** `srcnat` (Source NAT / Static IP)
* **Action:** `masquerade` (Dynamic IP)
* **Function:** Dynamically translates our private IP addresses into a single public IP address so that devices on the local LAN can access the WAN network.
`Setting IP > Firewall > NAT > +`

##  2. Ubuntu Server Target Configuration 🐧

The target server is intentionally built with several configuration flaws and vulnerabilities. This setup allows us to practice exploitation techniques and system hardening concurrently. The lab architecture is designed to evolve across different versions; every new release introduces updated software patches alongside brand-new vulnerabilities, rendering older exploit scripts obsolete.

### 🛠️ Assigning Static IP Address (Netplan)
Network configurations are written directly into the `/etc` directory to ensure that the static IP settings persist across system reboots. We will also explicitly bring up the newly added network interface link.


## Attack Vector Analysis & Exploitation Pathways
This section breaks down the tactical offensive kill-chain designed into the v1.0.0 baseline setup, showing how an attacker can chain these flaws together to compromise the target server.

  1. Information Gathering via Anonymous FTP Access: Attackers can authenticate using the default anonymous username without a password to explore the /var/ftp/pub/ directory. By downloading admin.txt, they can harvest leaked configuration logs or      plaintext operational notes.

  2. Unauthenticated Data Harvesting (Samba Public Share Traversal): The lack of restricted access control parameters allows any external entity to map, read, and write unauthorized files to the public SMB storage share, exposing corporate
     internal blueprints.

  4. Authentication Bypass via SQL Injection on index.php: Because input fields lack sanitization or parameterized queries, attackers can manipulate the backend database query logic. Injecting a classic boolean-based payload like ' OR '1'='1
     tricks the application into authenticating the session without requiring a valid username or password.

  6. Remote Code Execution via Arbitrary File Upload on dashboard.php: Once inside the administrative panel, the file upload mechanism acts as the ultimate entry point. Since the backend code does not filter extensions, an attacker can upload a
     malicious PHP web shell script. By directly browsing the file path in the uploads/ directory, they can execute system commands and trigger a Reverse Shell to gain full root-level control over the Ubuntu Server ecosystem.
